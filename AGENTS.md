# DeployHub 项目修改规则

## 1. 概述

本文档定义了 DeployHub 项目的代码修改规范和流程，确保团队协作高效、代码质量可控、文档同步更新。

## 2. 基本修改规则

### 2.1 修改前准备

- **需求确认**：修改前必须明确需求来源（Issue、PRD、口头沟通记录）
- **影响评估**：评估修改对现有功能、API、数据库的影响范围
- **分支管理**：从 `develop` 分支创建特性分支，分支命名规范：`feature/xxx` 或 `bugfix/xxx`

### 2.2 数据库安全规则（强制）

- **禁止操作范围**：严禁修改、删除、重置除 `deploy_hub` 数据库以外的任何数据库；严禁执行 `DROP DATABASE`、`docker-compose down -v` 等可能清除其他数据库数据的操作
- **MySQL服务保护**：禁止随意停止、重启或重置整个MySQL服务实例，避免影响其他业务数据库
- **数据隔离**：所有数据库操作必须限定在 `deploy_hub` 数据库内，使用完整表名（如 `deploy_hub.t_app`）
- **备份要求**：涉及数据变更的操作前，必须先确认仅影响 `deploy_hub` 数据库

### 2.2 代码规范

- **命名规范**：遵循 Java 命名规范，类名采用 PascalCase，方法名和变量名采用 camelCase
- **代码风格**：使用 4 空格缩进，每行不超过 120 字符
- **注释要求**：关键逻辑必须有注释说明，公共 API 需添加 Javadoc 注释
- **异常处理**：统一使用项目自定义异常类，避免直接抛出 RuntimeException

### 2.3 提交规范

每次提交需遵循以下格式：

```
[type]: [description]

[optional body]
```

**type 类型**：
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式调整（不影响功能）
- `refactor`: 代码重构
- `test`: 测试用例
- `chore`: 构建/工具相关

### 2.4 测试要求

- **单元测试**：新增功能必须编写单元测试，覆盖率不低于 80%
- **集成测试**：涉及多模块交互的修改需编写集成测试
- **回归测试**：修改前需运行现有测试套件确保不破坏原有功能

## 3. 强制同步规则

### 3.1 规则说明

**强制规则**：任何代码或功能修改完成后，必须同步更新以下文档：

| 文档 | 路径 | 更新内容 |
|------|------|----------|
| README.MD | `/README.MD` | 功能介绍、使用方法、快速开始 |
| 操作手册.MD | `/操作手册.MD` | 详细操作指南、配置说明、部署步骤 |

### 3.2 同步流程

```
代码修改完成 → 功能验证通过 → 更新 README.MD → 更新操作手册.MD → 提交代码
```

### 3.3 检查清单

提交代码前必须确认：

- [ ] README.MD 已同步更新（新增功能说明、修改使用方法、更新示例）
- [ ] 操作手册.MD 已同步更新（更新操作步骤、配置说明、部署指南）
- [ ] 文档版本号已更新（如有版本控制）
- [ ] 文档内容与代码实现一致

### 3.4 违规处理

- 首次违规：口头提醒，要求补充文档更新
- 再次违规：代码审核不通过，需完成文档同步后重新提交
- 多次违规：纳入绩效考核

## 4. 代码审核流程

### 4.1 审核要点

- 代码逻辑正确性
- 代码规范符合度
- 测试覆盖完整性
- 文档同步完成度（强制检查）
- 性能影响评估

### 4.2 审核通过条件

必须同时满足：

1. 至少 1 位代码所有者审核通过
2. 所有测试用例通过
3. README.MD 和操作手册.MD 已同步更新
4. 无重大代码质量问题

## 5. 文档维护职责

| 角色 | 职责 |
|------|------|
| 开发人员 | 负责同步更新相关文档 |
| 技术负责人 | 审核文档更新完整性 |
| QA 测试人员 | 验证文档与实际功能一致性 |

## 6. 版本控制

- 文档与代码使用同一仓库管理
- 文档更新与代码修改在同一提交中完成
- 文档版本与代码版本保持一致
